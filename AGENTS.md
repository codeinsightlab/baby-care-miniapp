# AGENTS.md

本文件定义 `baby-care-miniapp` uni-app 微信小程序端的一期开发规则。所有 AI / Codex / GPT 参与的小程序端实现、审查、修复都必须遵守本文件约束。

## 一、职责边界

`baby-care-miniapp` 技术栈为 uni-app，只负责微信小程序端页面、表单、状态展示和后端 API 交互，包括：

- 用户登录入口
- 宝宝选择与宝宝协作入口
- 护理记录录入
- 今日护理展示
- 历史记录查看
- 提醒节点设置
- 订阅消息授权触发
- 语音备注录入
- 与后端 API 交互

小程序端不负责最终业务归属判定。关键归属关系必须由后端按内部归属模型校验，包括：

- 当前用户是否属于宝宝协作空间
- 当前宝宝是否属于该协作空间
- 当前记录是否属于该协作空间
- 当前提醒节点是否属于当前用户或协作空间

禁止在小程序端写死业务归属，禁止把前端选择或缓存当作可信业务依据。

## 二、宝宝协作产品语义

小程序用户可见语义统一围绕“宝宝”和“照顾人”，多人共同查看和管理宝宝数据时使用以下表达：

- 宝宝协作
- 邀请照顾人
- 一起照顾宝宝
- 共享宝宝护理数据
- 共同查看记录
- 按照顾人设置提醒
- 已加入的照顾人

小程序用户可见文案禁止出现以下重概念：

- 家庭
- 家庭成员
- 加入家庭
- 创建家庭
- 家庭管理

后端可以继续使用 `family` / `family_member` 作为内部数据归属模型，但小程序页面、service view model、接口字段和用户提示应转换为 `collaboration` / `collaborator` 语义。后端 `OWNER` 对前端展示为“主要照顾人”，`MEMBER` 对前端展示为“照顾人”。

## 三、当前用户 / 宝宝协作 / 当前宝宝状态

- 当前用户必须来自微信登录后的后端用户信息。
- 当前宝宝必须来自后端返回的可访问宝宝列表。
- 当前宝宝是小程序主业务状态；底层协作空间只作为共享数据和权限边界，不作为用户侧主概念。
- 小程序可以缓存当前宝宝等选择，用于改善体验。
- 缓存只能作为界面默认值，不能作为可信业务依据。
- 请求接口时可以携带 `familyId`、`babyId`，但后端必须二次校验；普通业务优先用当前 `babyId` 查询。
- 切换宝宝后，今日页面、护理记录、历史记录、提醒节点、提醒执行、当前提醒与计划相关数据必须刷新。
- 后续实现“邀请照顾人 / 宝宝协作”时，前端不得把底层 `family` 暴露为家庭切换或家庭管理流程。

## 四、页面开发规则

一期页面保持简单，优先跑通闭环。页面优先级如下：

1. 登录页或登录入口
2. 宝宝创建 / 宝宝选择页
3. 宝宝档案页
4. 今日页
5. 护理记录新增页
6. 历史记录页
7. 计划与提醒页
8. 语音备注组件

开发要求：

- 不要一开始追求复杂首页。
- 不要一次性生成所有复杂交互。
- 简单表单可以一次生成。
- 今日页、提醒页、语音组件必须单独处理和验收。
- 不新增超出一期目标的页面。

## 五、小程序原型参考规则

后续实现 `baby-care-miniapp` 中的 uni-app 页面时，必须优先参考以下目录中的项目原型：

`/Users/user/Documents/mia/baby-care/pdf`

实现页面前必须先确认：

- 该功能是否在原型 PDF 中有对应页面。
- 对应页面的布局结构。
- 页面主要入口和跳转关系。
- 表单字段和按钮位置。
- 主要视觉风格。
- 是否存在弹窗、确认框、空状态、列表状态等交互。

如果原型中有明确设计，必须优先按原型实现。

如果原型中没有明确设计，但需求文档明确存在该功能，应保持与已有原型风格一致补齐，并在输出中说明“原型未覆盖，按现有风格补齐”。

如果原型和需求文档存在冲突，必须先停止并向用户确认，不允许自行取舍。

禁止：

- 脱离原型自由设计页面。
- 未查看原型就直接生成小程序页面。
- 为了炫技加入复杂动效、复杂组件或品牌级 UI 定制。
- 移动、重命名、删除原型 PDF 文件。

简单表单页面可以按原型批量生成。今日页、计划提醒页、历史记录页、语音备注交互等复杂页面，必须先确认原型结构再实现。

修改或实现页面后，必须更新 `docs/page-list.md` 或 `docs/prototype-reference.md`，记录页面与原型的对应关系。

## 六、护理记录规则

一期护理记录类型包括：

- 喂养
- 睡眠
- 换尿布 / 排泄
- 基础护理
- 运动与互动

小程序端只负责表单采集和展示。护理记录保存后以后端返回结果为准。

禁止：

- 在小程序端写死护理记录归属。
- 使用 mock 数据冒充真实接口。
- 在生产代码中保留测试数据。
- 在前端自行判断最终业务归属。

## 七、计划与提醒规则

- 固定计划模板来自后端。
- 用户调整的是自己的提醒节点。
- 小程序端不能修改系统计划模板。
- 小程序端只能调用接口启用、关闭、调整、新增、删除用户提醒节点。
- 当前一期完成提醒只更新或写入提醒执行记录，不自动创建护理记录；如后续要关联护理记录，必须先更新接口协议和状态流转文档。

禁止在小程序端实现复杂工作流、复杂重复规则或规则引擎。一期只按后端提供的提醒能力展示和操作。

## 八、微信订阅消息规则

- 小程序端只负责触发订阅授权。
- 授权结果需要传给后端记录。
- 订阅消息是否可发送、是否送达，以微信平台限制和后端发送结果为准。
- 不承诺 App 闹钟级强提醒。

禁止写出“确保提醒送达”“保证准时提醒”等描述。

## 九、语音备注规则

一期语音能力只用于备注输入。流程为：

用户点击语音按钮 -> 录音/识别 -> 返回文字 -> 填入备注 -> 用户确认 -> 保存记录

如果识别失败，应允许用户手动输入备注。

禁止：

- 自动解析奶量。
- 自动解析时间。
- 自动判断护理类型。
- 自动生成多条护理记录。
- AI 语义理解。
- 智能育儿分析。
- 医学分析或专业医疗建议。

## 十、API 交互规则

- API 请求必须统一封装。
- 不在页面中散落请求地址。
- `token` / `session` 必须统一管理。
- 接口错误要有基本提示。
- 登录失效要统一处理。
- 不要在页面内硬编码后端域名。
- 开发环境和生产环境地址要分开配置。
- 接口前缀统一使用 `/api`。
- 登录流程为 uni-app 获取微信 `code`，发送给后端换取登录 `token`。
- 小程序端保存 `token`，后续请求统一携带 `token`。
- 小程序登录只做 `openid` 与 `bc_user` 绑定，不做手机号绑定登录。
- 所有环境主链路均为无感登录；`miniappDevLoginEnabled = true` 时通过 `dev-login` 无感换取 token，`miniappDevLoginEnabled = false` 时通过 `wx.login + mini-login` 无感换取 token，登录通道只由 `config/env.js` 门禁决定。
- `login` 页只能作为 dev/test 辅助或异常兜底页，不作为生产主链路；禁止在生产环境使用 `dev-login`，禁止绕过 `config/env.js` 写死登录方式。
- `401` 表示 token 失效，可清 token / user 并按同一无感登录链路重试一次；`5xx`、网络异常、超时不清 token、不清 `BC_CURRENT_BABY_ID`，不强跳 login / splash / baby。
- `BC_CURRENT_BABY_ID` 只在用户主动选择 / 切换，或后端明确返回当前宝宝无权限 / 不存在时清理；服务异常、后端重启中、普通业务接口失败不得清理。
- App / Splash 启动阶段只做登录态和当前宝宝入口恢复，不无条件调用今日汇总、今日提醒等首页业务接口；业务数据由对应 tab 页面自行拉取并展示可重试错误态。
- 遇到接口缺失时先说明需要后端接口，不要自行编造接口语义。

## 十一、小程序 UI / 交互约束

小程序页面实现和修改必须遵守 `docs/ui-guidelines.md`。本节只记录稳定执行边界，具体页面 UI、样式、组件、数据展示和交互规则以 `docs/ui-guidelines.md` 为准。

- 原型优先。实现页面前必须先查看 `/Users/user/Documents/mia/baby-care/pdf` 下是否存在对应原型。
- 不引入新 UI 框架、外部 CSS 框架或新组件库。
- 不做复杂动效系统，只保留必要的加载、提交、确认和状态反馈。
- 不提前抽象组件；至少两个页面明确复用后，再考虑抽取组件。
- 不写 mock 业务数据，不用假数据冒充真实接口返回。
- 样式优先局部化，命名必须语义化，不随意污染全局样式，不随意修改 `App.vue` 全局样式。
- 页面变更必须同步更新对应 `docs/` 文档；页面清单或原型对应关系变化必须更新 `docs/page-list.md`。
- `family` 不作为用户侧主语义，普通护理业务页面不以家庭空间作为主对象。
- 用户可见多人协作入口必须使用“宝宝协作 / 邀请照顾人 / 照顾人”等表达，不得使用“家庭 / 家庭成员 / 加入家庭 / 创建家庭 / 家庭管理”。
- 当前宝宝是用户侧主状态，今日、记录、历史、提醒和语音备注等页面应围绕当前宝宝展开。
- 不允许把 UI 任务扩展成业务功能任务，不允许顺手重构小程序整体结构。
- today 首页必须表达“当前护理节奏流”，主线是“当前待执行 / 现在要照护什么”；页面应形成“当前宝宝 -> 当前待执行主卡 -> Quick Actions -> 今日轻量 Timeline”的自然节奏，禁止所有 section 同等级堆叠、补偿列表化或后台 dashboard 感。
- Reminder MVP 页面职责最终边界为：Today 负责当前护理节奏流，Reminder 负责历史未处理补偿流，Record 负责 CareRecord 护理事实历史，Plan 负责手动护理节奏配置，Baby 负责宝宝对象和协作关系；禁止页面再次膨胀成全能护理入口。
- today 首页允许 Care Status 状态表达，用于说明“宝宝现在是什么状态”；状态只能来自 record、plan、timeline event 和当前时间，禁止 AI 建议、健康分析、自动育儿建议、推荐系统、黑盒推断或复杂状态机。
- 今日时间轴必须展示“宝宝今日照护事件流”，只能消费统一 Timeline Event view model；禁止直接把 record list、reminder log 或按钮操作流水当作时间轴渲染。
- 今日时间轴必须保留真实事件，禁止自动合并、自动删除、语义压缩或 AI 摘要；允许通过 `eventType` 展示映射弱化 `PLAN_COMPLETED`、`PLAN_DELAYED`、`SYSTEM_EVENT` 等辅助事件，但不得改变事件真实性。
- 小程序按钮必须遵守 Button System Rules：主按钮使用实底、高对比文字、清晰边界和明确 active 态；次按钮可以轻量但必须有清晰描边；禁止半透明灰按钮、浅灰虚边、低对比 CTA 和依赖 uni-button 默认虚边。
- Reminder MVP 的 `ReminderCard`、`TimelineItem`、`QuickRecordSheet` 复用边界不得被页面复制替代；Today / Reminder 只能通过组件 `mode` 区分页面语义。

## 十二、小程序页面刷新策略

小程序页面刷新必须遵守项目级 `docs/ui-refresh-strategy.md`。适用页面包括 `today`、`reminder`、`record`、`plan`、`baby`、`history` 和 dashboard 类页面。

稳定规则：

- 页面刷新采用 stale-while-revalidate 风格：先保留最后一次成功显示数据，再后台刷新，成功后原子替换。
- 页面数据应拆分为 `displayState` 和 `refreshingState`，或使用等价的显示态与刷新态命名。
- `displayState` 表示当前可见 UI 的最后一次成功数据，请求期间不得清空。
- `refreshingState` 必须按 section 拆分，例如当前宝宝、当前待执行、时间轴、统计、列表、图表等。
- 请求失败时保留旧 UI，仅做 toast、局部错误提示或重试入口。
- 首次进入且没有 `displayState` 时，才允许 skeleton、初始化 loading 或错误态。
- 已有 `displayState` 后，刷新期间禁止整页 skeleton、整页 loading 或主体页面重建。

明确禁止：

- 请求前 `data = null`。
- 请求前 `list = []`。
- 请求前 reset 整个页面 state。
- 使用一个全局 `loading` 控制整个页面。
- 使用 `v-if="!loading"` 包裹整个页面主体。
- loading 期间销毁 swiper、scroll-view、timeline、card、chart 或表单容器。
- 为普通页面刷新引入 pinia、vuex、事件总线、复杂缓存系统、request manager 或全局请求编排。

竞态提醒：

- 后续涉及 `onShow`、宝宝切换、下拉刷新、tab 快速切换时，必须评估旧请求晚返回覆盖新数据的风险。
- 当前阶段优先使用页面局部最小保护，例如请求序号或目标 `babyId` 校验；不要提前扩展成全局请求系统。

## 十三、H5 dev 开发与视觉验收约束

- 后续 `baby-care-miniapp` 前端开发统一以 dev 环境为主，`config/env.js` 默认保持 `ENV = 'dev'`。
- 常规页面调试、交互检查和视觉验收以浏览器地址 `http://localhost:8081/#/` 为主。
- Codex 做前端视觉验收时，优先打开 `http://localhost:8081/#/` 及对应 hash 路由，不默认编译到微信小程序。
- 除非用户明确要求，不使用 mp-weixin 构建产物、微信开发者工具或真机作为普通前端视觉验收入口。
- 真实微信登录、订阅消息、微信专有 API 等能力需要单独说明验证条件，不能用 H5 视觉验收结论替代微信环境验收。
- 前端服务默认由用户保持 `8081` 常驻；Codex 只访问用户已有服务，不自行启动前端服务或更换端口。

## 十五、小程序埋点开发约束

当前阶段只建立埋点规范和统一入口，不开发真实埋点平台，不做真实数据上报。

- 后续小程序页面如需埋点，必须使用 `utils/tracker.js` 暴露的统一 `track()`。
- 不允许页面内随意 `console.log` 作为埋点。
- 不允许页面直接发埋点请求。
- 不允许页面直接调用第三方 analytics SDK。
- 不允许调用 `wx.reportAnalytics`。
- 不允许采集 `token`、`openid`、手机号、宝宝姓名、备注内容、语音识别文本等敏感字段。
- `data-track` 命名必须符合 `docs/tracking-spec.md` 的事件命名规范。
- 当前阶段埋点只允许 console 验证，不真实上报。
- 如果需要新增事件，必须先更新 `docs/tracking-spec.md`。
- Codex 实现页面时不得自行发明事件名。

## 十六、分层职责与异常边界约束

必须严格保持职责边界：

- Controller：只处理请求接收、参数校验、响应返回
- Service：负责业务编排、业务规则、事务边界
- Gateway / Client：只负责外部系统调用与结果适配
- DAO / Mapper：只负责数据读写
- 全局异常处理器：统一处理异常、日志、错误响应

禁止：

- Controller 写业务逻辑
- Controller 使用 try/catch 吞异常或兜底返回成功
- Controller 捕获异常后导致异常信息丢失
- Gateway / Client 写业务判断
- DAO / Mapper 承载业务语义
- DTO / VO / Entity 混用
- 多层重复转换
- 多处重复处理同一种异常

异常处理原则：

- 业务异常在 Service 层明确抛出
- 系统异常不在 Controller 层兜底吞掉
- Controller 不做异常降级
- 所有异常统一交给全局异常处理器处理
- 需要补充上下文时，可以包装异常，但不能丢失原始 cause

禁止无意义空值兜底：

- 对必填请求对象、必填字段，不允许使用 null-safe 写法掩盖错误
- 禁止类似：`String code = request == null ? null : request.getCode();`
- 如果 request 按契约必填，应直接使用 `request.getCode()` 或先显式校验 request 非空
- 如果 code 按契约必填，应显式校验 code 非空
- 不允许把“非法输入”转换成 null 后继续流转
- Codex 不得自动给所有按钮加埋点，只能按明确需求或规范中的关键事件加。
- 不允许自动采集所有点击、输入内容、页面曝光、停留时长或滚动深度。

## 十七、前端字段协议、骨架与 Mock 约束

- 构建页面骨架时必须先定义稳定的前端视图字段协议，明确页面 view model 需要哪些字段、字段语义和展示用途。
- 前端视图字段协议是页面消费协议，不是数据库字段设计；禁止写成“前端决定数据库字段”，也禁止要求页面直接消费数据库字段。
- 骨架阶段允许使用 mock 数据，但 mock 数据必须基于前端视图字段协议提供，字段命名不能随意发明。
- mock 数据只允许作为后端未接入前的临时数据源，不允许冒充真实接口已完成，不允许保留为生产业务数据。
- 页面组件禁止直接散落 mock 字段转换逻辑。
- 必须通过 service / adapter / mapper 等中间层承接 mock 和真实接口数据：
  - api 层负责请求。
  - service / adapter 层负责数据来源切换、字段适配和 view model 组装。
  - 页面只消费 service / adapter 输出后的稳定 view model。
- 后续接入真实后端时，只允许优先替换 service / adapter 内部数据来源或转换逻辑，页面结构和字段消费方式应尽量不改。
- 如果页面展示字段与接口字段语义不同，必须在 service / adapter 层明确转换，并说明来源字段和视图字段的对应关系。
- 禁止页面直接依赖随意命名字段，例如页面自己写 `babyName2`、mock 写 `titleText`，导致后续接接口时大面积返工。
- Codex 每次新增骨架页、mock 数据或静态数据时，回执必须说明：
  - mock 数据位置。
  - 前端视图字段协议。
  - 页面是否通过 service / adapter 消费稳定 view model。
  - 后续接真实接口的替换点。

## 十八、Codex 执行规则

- 修改小程序前先说明计划修改的文件。
- 一次任务只完成一个明确闭环。
- 不顺手重构无关页面。
- 不新增超出需求的页面。
- 不新增无关依赖。
- 不把接口 mock 留在生产代码。
- 不把后端实现规则写进小程序端规则。
- 不把小程序用户和 RuoYi 后台管理员混为一类用户。
- 不在未查看原型的情况下直接实现页面。
- 遇到接口缺失时先说明需要后端接口，不要自行编造接口语义。

### 小程序测试约束

- 新增或修改小程序功能时，必须同步考虑测试代码和必要边界测试，不能只写页面或 service 不写测试。
- 前端新增 / 修改功能后，必须同步补函数级回归、页面逻辑级回归，并更新 `docs/frontend-regression-index.md`。
- 函数级回归和页面逻辑级回归共同作为编译前强门禁；涉及本次变更的强门禁测试必须能在编译前快速执行。
- 函数级回归必须覆盖 utils、service、enum、API 参数构造、payload 构造、数据转换函数、scene / token 解析、request / response normalize 等低成本稳定逻辑。
- 页面逻辑级回归不要求测试 Vue 样式，但必须覆盖关键页面状态判断、启动分流、token 生命周期、pending token 一次性消费、refreshingState / displayState、删除后状态变化、启停状态、无 token fallback 和页面入口边界。
- 复杂页面逻辑不得长期藏在不可测试的 `onLoad` / `onShow` 中；新增或调整启动分流、token 生命周期、刷新状态、枚举映射、payload 组装时，优先抽成可测试函数或轻量 Node 测试脚本。
- 前端测试优先覆盖 service、api、纯函数、枚举映射、字段适配和参数构造；页面 UI 不强制引入大型测试框架。
- 边界测试应按本次功能实际涉及范围覆盖：空参数、非法枚举、接口失败、重复提交、启停状态、删除后刷新不可见，以及涉及 `babyId` / `familyId` / `userId` 时的权限 / 归属边界。
- 对页面刷新类功能，必须关注接口失败时是否保留旧 UI、重复刷新是否覆盖新数据、删除或启停失败是否不会伪造成功状态。
- `node --check` 只表示 JavaScript 语法检查，不是业务回归、页面逻辑验证或功能测试；禁止用 `node --check` 冒充功能测试。
- 静态搜索、`node --check` 只能作为辅助质量检查，不能替代关键 service / api / 纯函数业务测试；轻量 node 回归脚本只有在包含明确断言和业务边界时才可作为对应模块回归。
- 浏览器自动化只作为人工或阶段性验收手段，不作为编译前门禁，不进入每次构建强制链路；不为此引入 Playwright / Cypress 等浏览器自动化框架。
- 不为了测试引入复杂依赖或大型测试框架；优先采用项目现有体系或最小可用 node 测试脚本。
- 如果本轮暂时不能补测试，必须在 `log/codex-task-log.md` 或最终回执中说明原因和后续补测点。
- 后续新增 / 修改模块时，必须同步维护 `docs/frontend-regression-index.md`，登记模块、回归文件、覆盖边界、执行命令、门禁级别和当前缺口。

## 十九、本地前端服务访问规则

- 小程序端联调或页面验收时，前端服务默认由用户保持运行。
- 默认访问地址固定为 `http://localhost:8081/`，页面路由访问使用 `http://localhost:8081/#/` 及对应 hash 路由。
- Codex 不得自行启动前端服务。
- Codex 不得更换前端端口。
- 如需页面验证、视觉检查或路由可达性检查，只访问用户已有的 `8081` 服务。
- 如果 `8081` 不可达，只记录不可达原因，不自行启动服务。
- 禁止随意占用新端口或同时启动多个重复 dev server。
- 禁止修改接口地址、路由结构或业务配置，只为适配本地临时端口。
- 联调后端时，后端服务默认由用户保持 `http://localhost:8080/` 常驻；如不确定后端端口，先按根目录 `AGENTS.md` 的前后端服务运行与验证协作规则检查后端配置。

## 二十、小程序文档同步要求

小程序每次实现功能、修改功能或调整页面业务逻辑后，必须同步更新对应文档。

- 当前用户、宝宝协作、当前宝宝状态逻辑变化：更新 `docs/miniapp-state.md`。
- 接口请求参数、返回结构、登录态处理变化：更新 `docs/api-contract.md`。
- 提醒授权、订阅消息触发、稍后提醒变化：更新 `docs/reminder-flow.md`。
- 语音备注录入流程变化：更新 `docs/voice-input.md`。
- 页面清单或页面主流程变化：更新 `docs/page-list.md`。
- 页面与原型对应关系变化：更新 `docs/prototype-reference.md` 或 `docs/page-list.md`。
- 小程序埋点事件、payload 边界或接入方式变化：更新 `docs/tracking-spec.md`。
- 重要页面流程变更：更新 `docs/changelog.md`。
- 稳定开发规则、页面职责边界或禁止事项变化：更新本文件。

禁止只修改小程序代码不更新文档。禁止让后续开发者或 Codex 只能通过重新阅读页面代码来理解已实现逻辑。
