# AGENTS.md

本文件定义 `baby-care-miniapp` uni-app 微信小程序端的一期开发规则。所有 AI / Codex / GPT 参与的小程序端实现、审查、修复都必须遵守本文件约束。

## 一、职责边界

`baby-care-miniapp` 技术栈为 uni-app，只负责微信小程序端页面、表单、状态展示和后端 API 交互，包括：

- 用户登录入口
- 家庭与宝宝选择
- 护理记录录入
- 今日护理展示
- 历史记录查看
- 提醒节点设置
- 订阅消息授权触发
- 语音备注录入
- 与后端 API 交互

小程序端不负责最终业务归属判定。关键归属关系必须由后端校验，包括：

- 当前用户是否属于该家庭
- 当前宝宝是否属于该家庭
- 当前记录是否属于当前家庭
- 当前提醒节点是否属于当前用户或家庭

禁止在小程序端写死业务归属，禁止把前端选择或缓存当作可信业务依据。

## 二、当前用户 / 当前家庭 / 当前宝宝状态

- 当前用户必须来自微信登录后的后端用户信息。
- 当前家庭必须来自后端返回的家庭列表。
- 当前宝宝必须来自当前家庭下的宝宝列表。
- 当前宝宝是小程序主业务状态，当前家庭是共享空间和权限边界。
- 小程序可以缓存当前家庭、当前宝宝等选择，用于改善体验。
- 缓存只能作为界面默认值，不能作为可信业务依据。
- 请求接口时可以携带 `familyId`、`babyId`，但后端必须二次校验；普通业务优先用当前 `babyId` 查询。
- 切换家庭或宝宝后，今日页面、护理记录、历史记录、提醒节点、提醒执行、当前提醒与计划相关数据必须刷新。
- 切换家庭本质上更新可访问宝宝集合，不作为普通护理业务主查询入口。

## 三、页面开发规则

一期页面保持简单，优先跑通闭环。页面优先级如下：

1. 登录页或登录入口
2. 家庭创建 / 加入页
3. 宝宝档案页
4. 今日页
5. 护理记录新增页
6. 历史记录页
7. 计划与提醒页
8. 语音备注组件

开发要求：

- 不要一开始追求复杂首页。
- 不要一次性生成所有复杂交互。
- 简单表单可以一次生成。
- 今日页、提醒页、语音组件必须单独处理和验收。
- 不新增超出一期目标的页面。

## 四、小程序原型参考规则

后续实现 `baby-care-miniapp` 中的 uni-app 页面时，必须优先参考以下目录中的项目原型：

`/Users/user/Documents/mia/baby-care/pdf`

实现页面前必须先确认：

- 该功能是否在原型 PDF 中有对应页面。
- 对应页面的布局结构。
- 页面主要入口和跳转关系。
- 表单字段和按钮位置。
- 主要视觉风格。
- 是否存在弹窗、确认框、空状态、列表状态等交互。

如果原型中有明确设计，必须优先按原型实现。

如果原型中没有明确设计，但需求文档明确存在该功能，应保持与已有原型风格一致补齐，并在输出中说明“原型未覆盖，按现有风格补齐”。

如果原型和需求文档存在冲突，必须先停止并向用户确认，不允许自行取舍。

禁止：

- 脱离原型自由设计页面。
- 未查看原型就直接生成小程序页面。
- 为了炫技加入复杂动效、复杂组件或品牌级 UI 定制。
- 移动、重命名、删除原型 PDF 文件。

简单表单页面可以按原型批量生成。今日页、计划提醒页、历史记录页、语音备注交互等复杂页面，必须先确认原型结构再实现。

修改或实现页面后，必须更新 `docs/page-list.md` 或 `docs/prototype-reference.md`，记录页面与原型的对应关系。

## 五、护理记录规则

一期护理记录类型包括：

- 喂养
- 睡眠
- 换尿布 / 排泄
- 基础护理
- 运动与互动

小程序端只负责表单采集和展示。护理记录保存后以后端返回结果为准。

禁止：

- 在小程序端写死护理记录归属。
- 使用 mock 数据冒充真实接口。
- 在生产代码中保留测试数据。
- 在前端自行判断最终业务归属。

## 六、计划与提醒规则

- 固定计划模板来自后端。
- 用户调整的是自己的提醒节点。
- 小程序端不能修改系统计划模板。
- 小程序端只能调用接口启用、关闭、调整、新增、删除用户提醒节点。
- 提醒完成后应创建护理记录或关联对应记录，具体以接口设计为准。

禁止在小程序端实现复杂工作流、复杂重复规则或规则引擎。一期只按后端提供的提醒能力展示和操作。

## 七、微信订阅消息规则

- 小程序端只负责触发订阅授权。
- 授权结果需要传给后端记录。
- 订阅消息是否可发送、是否送达，以微信平台限制和后端发送结果为准。
- 不承诺 App 闹钟级强提醒。

禁止写出“确保提醒送达”“保证准时提醒”等描述。

## 八、语音备注规则

一期语音能力只用于备注输入。流程为：

用户点击语音按钮 -> 录音/识别 -> 返回文字 -> 填入备注 -> 用户确认 -> 保存记录

如果识别失败，应允许用户手动输入备注。

禁止：

- 自动解析奶量。
- 自动解析时间。
- 自动判断护理类型。
- 自动生成多条护理记录。
- AI 语义理解。
- 智能育儿分析。
- 医学分析或专业医疗建议。

## 九、API 交互规则

- API 请求必须统一封装。
- 不在页面中散落请求地址。
- `token` / `session` 必须统一管理。
- 接口错误要有基本提示。
- 登录失效要统一处理。
- 不要在页面内硬编码后端域名。
- 开发环境和生产环境地址要分开配置。
- 接口前缀统一使用 `/api`。
- 登录流程为 uni-app 获取微信 `code`，发送给后端换取登录 `token`。
- 小程序端保存 `token`，后续请求统一携带 `token`。
- 小程序登录只做 `openid` 与 `bc_user` 绑定，不做手机号绑定登录。
- 遇到接口缺失时先说明需要后端接口，不要自行编造接口语义。

## 十、小程序 UI / 交互约束

小程序页面实现和修改必须遵守 `docs/ui-guidelines.md`。本节只记录稳定执行边界，具体页面 UI、样式、组件、数据展示和交互规则以 `docs/ui-guidelines.md` 为准。

- 原型优先。实现页面前必须先查看 `/Users/user/Documents/mia/baby-care/pdf` 下是否存在对应原型。
- 不引入新 UI 框架、外部 CSS 框架或新组件库。
- 不做复杂动效系统，只保留必要的加载、提交、确认和状态反馈。
- 不提前抽象组件；至少两个页面明确复用后，再考虑抽取组件。
- 不写 mock 业务数据，不用假数据冒充真实接口返回。
- 样式优先局部化，命名必须语义化，不随意污染全局样式，不随意修改 `App.vue` 全局样式。
- 页面变更必须同步更新对应 `docs/` 文档；页面清单或原型对应关系变化必须更新 `docs/page-list.md`。
- `family` 不作为用户侧主语义，普通护理业务页面不以家庭空间作为主对象。
- 当前宝宝是用户侧主状态，今日、记录、历史、提醒和语音备注等页面应围绕当前宝宝展开。
- 不允许把 UI 任务扩展成业务功能任务，不允许顺手重构小程序整体结构。

## 十一、H5 dev 开发与视觉验收约束

- 后续 `baby-care-miniapp` 前端开发统一以 dev 环境为主，`config/env.js` 默认保持 `ENV = 'dev'`。
- 常规页面调试、交互检查和视觉验收以浏览器地址 `http://localhost:8081/#/` 为主。
- Codex 做前端视觉验收时，优先打开 `http://localhost:8081/#/` 及对应 hash 路由，不默认编译到微信小程序。
- 除非用户明确要求，不使用 mp-weixin 构建产物、微信开发者工具或真机作为普通前端视觉验收入口。
- 真实微信登录、订阅消息、微信专有 API 等能力需要单独说明验证条件，不能用 H5 视觉验收结论替代微信环境验收。
- 前端服务优先复用本地 `8081`，启动前必须先检查该端口是否已有目标服务。

## 十二、一期禁止范围

一期小程序端禁止实现或描述以下能力：

- AI 问答。
- AI 智能分析。
- 智能育儿建议。
- 医学分析。
- 发育异常判断。
- 专业医疗建议。
- 复杂运营后台。
- 多角色复杂权限。
- 自动化运营工具。
- App 级强提醒。
- 蓝牙设备接入。
- 智能硬件联动。
- 复杂自然语言解析。
- 自动从语音中提取奶量、时间、护理类型等业务字段。

## 十三、小程序埋点开发约束

当前阶段只建立埋点规范和统一入口，不开发真实埋点平台，不做真实数据上报。

- 后续小程序页面如需埋点，必须使用 `utils/tracker.js` 暴露的统一 `track()`。
- 不允许页面内随意 `console.log` 作为埋点。
- 不允许页面直接发埋点请求。
- 不允许页面直接调用第三方 analytics SDK。
- 不允许调用 `wx.reportAnalytics`。
- 不允许采集 `token`、`openid`、手机号、宝宝姓名、备注内容、语音识别文本等敏感字段。
- `data-track` 命名必须符合 `docs/tracking-spec.md` 的事件命名规范。
- 当前阶段埋点只允许 console 验证，不真实上报。
- 如果需要新增事件，必须先更新 `docs/tracking-spec.md`。
- Codex 实现页面时不得自行发明事件名。
- Codex 不得自动给所有按钮加埋点，只能按明确需求或规范中的关键事件加。
- 不允许自动采集所有点击、输入内容、页面曝光、停留时长或滚动深度。

## 十四、Codex 执行规则

- 修改小程序前先说明计划修改的文件。
- 一次任务只完成一个明确闭环。
- 不顺手重构无关页面。
- 不新增超出需求的页面。
- 不新增无关依赖。
- 不把接口 mock 留在生产代码。
- 不把后端实现规则写进小程序端规则。
- 不把小程序用户和 RuoYi 后台管理员混为一类用户。
- 不在未查看原型的情况下直接实现页面。
- 遇到接口缺失时先说明需要后端接口，不要自行编造接口语义。

## 十四、本地前端服务访问规则

小程序端联调或页面验收时，必须优先复用已有本地服务，不允许因为访问失败就立即新开前端服务。

- 优先访问已有本地前端端口。
- 如果无法确认前端端口，先查看 `package.json`、`vite.config.*`、`vue.config.js`、`manifest.json`、`pages.json`、HBuilderX / uni-app 运行日志、`README` 和 `docs/`。
- 如果确认没有目标前端服务在运行，才允许自行启动前端服务。
- 启动前端服务前必须说明原因。
- 禁止未检查已有前端端口就启动前端。
- 禁止随意占用新端口或同时启动多个重复 dev server。
- 禁止修改接口地址、路由结构或业务配置，只为适配本地临时端口。
- 联调后端时，默认优先访问本地 `8080`；如不确定后端端口，先按根目录 `AGENTS.md` 的本地服务访问规则检查后端配置。

## 十五、小程序文档同步要求

小程序每次实现功能、修改功能或调整页面业务逻辑后，必须同步更新对应文档。

- 当前用户、当前家庭、当前宝宝状态逻辑变化：更新 `docs/miniapp-state.md`。
- 接口请求参数、返回结构、登录态处理变化：更新 `docs/api-contract.md`。
- 提醒授权、订阅消息触发、稍后提醒变化：更新 `docs/reminder-flow.md`。
- 语音备注录入流程变化：更新 `docs/voice-input.md`。
- 页面清单或页面主流程变化：更新 `docs/page-list.md`。
- 页面与原型对应关系变化：更新 `docs/prototype-reference.md` 或 `docs/page-list.md`。
- 小程序埋点事件、payload 边界或接入方式变化：更新 `docs/tracking-spec.md`。
- 重要页面流程变更：更新 `docs/changelog.md`。
- 稳定开发规则、页面职责边界或禁止事项变化：更新本文件。

禁止只修改小程序代码不更新文档。禁止让后续开发者或 Codex 只能通过重新阅读页面代码来理解已实现逻辑。
